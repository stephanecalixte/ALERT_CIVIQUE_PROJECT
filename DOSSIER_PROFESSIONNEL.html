Introduction

Ce dossier professionnel présente le projet Alert_Civique, développé dans le cadre de la validation du Titre Professionnel Concepteur Développeur d'Applications (CDA, niveau 6 du Cadre National des Certifications Professionnelles — RNCP 37873).

Alert_Civique est une application mobile de signalement d'incidents civiques urgents, permettant à tout citoyen de déclencher en quelques secondes une alerte géolocalisée (SOS, agression, accident, incendie), d'enregistrer et partager une vidéo en direct (livestream), et de notifier automatiquement ses contacts de confiance. Une interface d'administration web permet aux équipes de traiter et résoudre les incidents en temps réel.

La plateforme repose sur une architecture multi-couches : une application mobile React Native (Expo) communicant avec une API REST Spring Boot (Java), complétée d'un serveur Node.js dédié à la gestion des médias vidéo, le tout persisté dans une base de données MySQL. Un tableau de bord d'administration React/Vite complète le dispositif.

Ce dossier couvre l'intégralité du cycle de vie du projet : analyse du besoin, conception technique, développement full-stack, sécurisation et tests, jusqu'à la livraison du prototype fonctionnel. Il est organisé en cinq parties auxquelles s'ajoutent des annexes techniques.

Partie 1 — Contexte et analyse stratégique

1.1 Présentation du projet Alert_Civique

En France, les dispositifs de signalement d'urgence citoyenne sont fragmentés entre les applications des services de secours (15, 17, 18), les applications municipales isolées et les réseaux sociaux. Aucune solution unifiée, mobile-first et accessible à tous ne permet à un citoyen de déclencher une alerte géolocalisée, de la documenter en vidéo et de prévenir instantanément ses proches — tout en un seul geste.

Alert_Civique comble ce vide en proposant :

  • Une application mobile native (Android / iOS via Expo) accessible immédiatement après une unique inscription
  • Quatre types d'alertes couvrant les urgences civiques : SOS, Agression, Accident, Incendie
  • Un signalement photo ou vidéo automatiquement géolocalisé, enregistré en galerie et transmis aux autorités
  • Un système de livestream vidéo associé à chaque incident, uploadé automatiquement
  • La gestion de contacts de confiance notifiés en cas d'alerte SOS
  • Un tableau de bord d'administration pour le suivi et la résolution des incidents

2.1 Expression des besoins

Besoins fonctionnels — Citoyens (application mobile)

  • NF1 — Inscription unique : formulaire d'inscription complet (nom, prénom, email, mot de passe sécurisé, téléphone, date de naissance) avec au moins un contact de confiance obligatoire. Après inscription, l'accès est permanent sans nouvelle authentification.

  • NF5 — Contacts de confiance : gestion CRUD des contacts (nom, email, téléphone), minimum 1 requis à l'inscription, possibilité d'en ajouter ultérieurement.

  • NF7 — Tableau de bord : statistiques temps réel (signalements, utilisateurs, livestreams, alertes SOS), graphiques de statuts, incidents actifs géolocalisés.

  • NF9 — Gestion des livestreams : liste des vidéos, lecture intégrée, suppression depuis le serveur média.
  • NF10 — Gestion des utilisateurs : liste, profil, suppression.

Flux de données — Exemple : signalement vidéo

  1. L'utilisateur appuie sur « Vidéo » dans l'app mobile
  2. L'app appelle POST /api/livestream/create sur Java → création de l'entrée en BDD (status = LIVE)
  3. L'appareil photo natif s'ouvre (via ImagePicker.launchCameraAsync)
  4. Après enregistrement, la vidéo est sauvegardée en galerie (MediaLibrary.saveToLibraryAsync)
  5. La vidéo est uploadée vers Node.js (POST /upload)
  6. Node.js retourne l'URL → l'app met à jour le livestream Java (PUT /api/livestream/update avec videoUrl)
  7. Node.js notifie le dashboard admin en temps réel (WebSocket)

3.2 Modèle de données (MCD / MLD)

Le modèle de données MySQL est centré sur l'entité Report (signalement), reliée aux utilisateurs, aux contacts de confiance, aux livestreams et aux messages. Toutes les clés primaires sont auto-incrémentées (BIGINT IDENTITY).

Règles de gestion

  • Un utilisateur doit avoir au moins un contact de confiance à l'inscription
  • Un signalement passe obligatoirement par le statut PENDING avant tout autre statut
  • Un livestream est créé au moment du démarrage de l'enregistrement (status = LIVE) puis mis à jour avec l'URL vidéo après upload
  • Le token d'activation de compte a une durée de vie limitée (app.jwt.activation-expiration-ms)
  • Les mots de passe sont hashés avec Argon2 (jamais stockés en clair)

3.4 Sécurité et conformité

Authentification et autorisation

  • JWT signé avec un secret externalisé en variable d'environnement (JWT_SECRET, JWS_SECRET)
  • Filtre Spring Security JwtAuthenticationFilter : validation du token à chaque requête protégée
  • Système RBAC (Role-Based Access Control) : ROLE_CLIENT, ROLE_ADMIN, ROLE_MODERATOR
  • Validation des données à l'inscription côté client (React Native) ET côté serveur (Jakarta @Valid, annotation @ValidPassword)
  • Activation du compte par email avec token à durée limitée

  • Mots de passe jamais stockés en clair (Argon2)

5.2 Difficultés rencontrées et solutions apportées

Difficulté 1 — Enregistrement vidéo sur Expo Go Android

Problème : L'API CameraView.recordAsync() d'Expo échouait systématiquement sur Expo Go Android avec une erreur liée aux permissions audio.

Solution : Remplacement complet par ImagePicker.launchCameraAsync({ mediaTypes: 'videos' }) qui délègue l'enregistrement à l'application caméra native du téléphone. Suppression de l'appel à MediaLibrary.requestPermissionsAsync() qui déclenchait un crash audio — l'appel à MediaLibrary.saveToLibraryAsync() se charge directement de la permission. Résultat : vidéos fonctionnelles et sauvegardées en galerie.

Difficulté 2 — Stale closure dans sendStopStream

Problème : Le hook useLiveStreamAPI capturait l'ID du livestream au moment de la création du callback, pas à son exécution. Si l'état React était mis à jour entre-temps, sendStopStream utilisait un ID obsolète (stale closure).

Solution : Introduction d'une useRef (currentLivestreamIdRef) mise à jour synchroniquement dans sendStartStream. Le callback sendStopStream lit toujours la valeur courante via currentLivestreamIdRef.current, contournant le problème de closure.

Difficulté 3 — Session utilisateur perdue (user: null)

Problème : Après inscription, l'objet utilisateur pouvait ne pas être stocké dans AsyncStorage si l'auto-login échouait (erreur réseau). Lors du redémarrage de l'application, user était null, rendant impossible l'ajout de contacts de confiance.

Solution : Implémentation d'un mécanisme de récupération silencieuse dans AuthContext : si @user est absent au démarrage mais que @credentials existent, un appel automatique à /api/auth/login restaure la session complète (token + objet utilisateur avec userId) sans interaction utilisateur.

Difficulté 4 — Contexte Spring Boot ne démarre pas en tests

Problème : Tous les tests JUnit échouaient avec IllegalState Failed to load ApplicationContext. Le fichier src/test/resources/application.properties remplaçait le fichier principal sans inclure la propriété node.media.base-url, injectée par @Value dans LiveStreamServiceImpl sans valeur par défaut.

Solution : Ajout de node.media.base-url=http://localhost:9091 dans src/test/resources/application.properties. Tous les tests passent désormais.

Difficulté 5 — Tab bar chevauchant la navigation Android

Problème : Sur Android, la barre d'onglets de l'application chevauchait la barre de navigation système (boutons retour/accueil).

Solution : Utilisation de useSafeAreaInsets pour appliquer dynamiquement paddingBottom: insets.bottom et height: TAB_BAR_HEIGHT + insets.bottom à la tab bar.

// AuthContext.tsx — Récupération automatique de session au démarrage
// Si @user est null mais @credentials existent → re-login silencieux
async function reLoginWithCredentials(creds) {
  const response = await fetch(`${JAVA_BASE_URL}/api/auth/login`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ username: creds.email, password: creds.password }),
  });
  if (!response.ok) return null;
  const data = await response.json();
  const userId = data.userId ?? data.id ?? 0;
  if (!userId) return null;
  return {
    token: data.token ?? data.accessToken ?? '',
    user: {
      userId,
      name: [data.firstname, data.lastname].filter(Boolean).join(' '),
      email: data.email ?? creds.email,
      isAdmin: data.isAdmin ?? false,
    },
  };
}

A.3 Front-end React Native — Correction stale closure avec useRef

// useLiveStreamApi.ts — Fix stale closure avec useRef
export function useLiveStreamAPI(token, userId) {
  const [currentLivestreamId, setCurrentLivestreamId] = useState(null);

  // useRef : toujours à jour, même dans les callbacks capturés par closure
  const currentLivestreamIdRef = useRef(null);

  const sendStartStream = useCallback(async (facing) => {
    const response = await LiveStreamService.sendLiveStreamData(payload, token);
    const id = response?.livestreamId || null;

    setCurrentLivestreamId(id);           // Mise à jour React state (async)
    currentLivestreamIdRef.current = id;  // Mise à jour ref (synchrone, toujours frais)
    return id;
  }, [userId, token]);

  const sendStopStream = useCallback(async () => {
    // Utilise la ref, pas le state — évite le stale closure
    const id = currentLivestreamIdRef.current;
    if (!id) return;
    await LiveStreamService.sendLiveStreamData({ livestreamId: id, ... }, token);
  }, [userId, token]);
}

Relations principales

  • users → reports : 1 utilisateur peut avoir N signalements (OneToMany)
  • users → lives_stream : 1 utilisateur peut avoir N livestreams (OneToMany)
  • users → trusted_contacts : 1 utilisateur peut avoir N contacts de confiance (OneToMany)
  • reports → report_messages : 1 signalement peut avoir N messages (OneToMany)
  • users ↔ roles : relation ManyToMany via table users_roles

TermeDéfinition
API RESTInterface de programmation basée sur HTTP, exposant des ressources sous format JSON via les méthodes GET/POST/PUT/DELETE.
JWT (JSON Web Token)Standard d'authentification stateless : token signé contenant les claims de l'utilisateur, validé à chaque requête sans session serveur.
Argon2Algorithme de hachage de mot de passe, vainqueur de la Password Hashing Competition (2015). Résistant aux attaques GPU et rainbow tables.
React Native / ExpoFramework JavaScript permettant de développer des applications mobiles natives iOS et Android à partir d'une base de code unique.
Spring BootFramework Java facilitant la création d'applications Spring autonomes et prêtes pour la production, avec configuration automatique.
Spring SecurityModule Spring gérant l'authentification, les autorisations (RBAC) et la protection contre les attaques courantes (CSRF, XSS).
JPA / HibernateJava Persistence API : standard Java de mapping objet-relationnel (ORM). Hibernate en est l'implémentation principale, permettant de manipuler la base MySQL via des objets Java.
AsyncStorageSystème de stockage clé-valeur asynchrone et persistant fourni par React Native, équivalent mobile de localStorage.
Stale ClosureProblème React où un callback capture une valeur d'état obsolète. Résolu par l'utilisation de useRef pour les valeurs devant toujours être fraîches.
expo-routerSystème de routage pour Expo basé sur la structure des fichiers (comme Next.js), permettant une navigation déclarative.
ImagePicker (Expo)Module Expo donnant accès à l'appareil photo natif du smartphone pour la capture photo ou vidéo.
